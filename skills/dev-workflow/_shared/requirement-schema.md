# Requirement Schema — `需求.md` / `改动.md` 字段契约

> **定位**：reference 文档，定义跨 skill 交接的稳定"接口"。`需求.md`（产品 → 开发 / 测试）与 `改动.md`（开发 → 测试）的字段 schema 在此固定；下游 skill 按正则或固定字段名定点解析。
>
> **强约束**：字段顺序固定，**不得乱序**；缺失字段以 `无` / `N/A` 占位，**不得删除**。

---

## 0 通用约定

### 0.1 语言

本 schema **默认中文章节标题**（如 `## 背景` / `## 接口变更` / `### FP-1 ...`）。下游所有解析正则都锚定中文章节名；切换到其他语言版本（如全英文 PRD）需要同步改本文件 + 全部下游 skill 的解析正则——目前**尚未支持多语言版本**。

产物文件名同样为中文（`需求.md` / `改动.md` / `测试.md` 等），见 [`./file-conventions.md`](./file-conventions.md) §1。

### 0.2 模板占位符规范

本文件及所有下游 skill 模板中：

- **`<尖括号>`** 表示必须替换为实际内容的占位符。例：`<编号>` / `<功能名称>` / `<原因描述>`。
- **禁止使用圆括号中文注释作为占位**（如 `（描述本需求的业务背景）`）—— 这种形式很容易被 Claude 误解为"指导文字应保留"或"需要照抄"，导致产出混入元说明。
- 模板顶部应统一声明：`> 以下所有 <尖括号> 必须替换为实际内容；不得照抄占位符本身。`

下游所有 skill 的模板（PRD.md / 需求.md / 可行性.md / 改动.md / 测试.md / 自审报告.md）必须遵守本规范。

## 1 `需求.md` 字段规范

### 1.1 字段顺序（固定，不得乱序）

```
(头部元信息)
# 需求 - {编号} - {功能名}
> 状态：...
> 优先级：...
> 工作量：...
> 前置依赖：...

## 背景
## 目标
## 功能点
    ### FP-1 {功能点名称}
    ### FP-2 ...
## 验收标准
## 非功能需求
## 影响面提示（产品视角）
```

**顺序固定**；下游 skill 按"章节标题 + 编号正则"解析，乱序会破坏 feasibility-assessor / change-summarizer / test-runner 的稳定性。**不得乱序**、**不得重命名章节**。

### 1.2 头部元信息

| 字段 | 取值 | 缺失处理 |
|------|------|----------|
| `状态` | `待开发` / `评估中` / `开发中` / `测试中` / `已完成`（与 [`./file-conventions.md`](./file-conventions.md) §4 表情符一致） | 初始必须为 `待开发` |
| `优先级` | `P0` / `P1` / `P2` | 缺失填 `P2` 并提示人工复核 |
| `工作量` | `小` / `中` / `大` | 未评估前填 `待评估` |
| `前置依赖` | 逗号分隔的子需求编号（如 `001, 003`）或 `无` | **必填**，无依赖时写 `无`，不得省略 |

### 1.3 `## 背景` / `## 目标`

- 背景：1 段自由文本（≤ 200 字建议）。
- 目标：无序列表 `- ...`，每条一个目标。
- 缺失一方：写 `无` 于该节之下。

### 1.4 `## 功能点`（核心字段，下游定点解析）

- 每个功能点独占三级标题，**格式严格**：

  ```
  ### FP-{N} {功能点名称}
  ```

  - 编号正则：`^### FP-(\d+)`（行首 + 三级标题 + 空格 + `FP-` + 数字 + 空格 + 名称）
  - N 从 1 起，连续递增，不得跳号。

- 每个 FP 下的子字段（顺序固定）：

  ```
  - 行为描述：...
  - 输入：...
  - 输出：...
  - 异常：...
  ```

  子字段必须全部保留。无内容写 `无`，例：`- 异常：无`。

### 1.5 `## 验收标准`

- 使用复选框格式：`- [ ] AC-{N} ...`。
- 编号正则：`AC-(\d+)`（行内出现即可）。
- N 从 1 起，连续递增。
- 未勾选 `[ ]` 表示未通过；`[x]` 表示已通过；下游 `test-runner` 按此列表生成功能测试用例。

### 1.6 `## 非功能需求`

- 无序列表，每条一类（性能 / 安全 / 兼容性 / 可维护性 / ...）。
- 无非功能需求写 `- 无`。

### 1.7 `## 影响面提示（产品视角）`

- 产品侧对"本需求可能触及的现有功能"的提示。
- 无序列表；下游 `feasibility-assessor` 会交叉验证，`test-runner` 回归测试会参考。
- 无则写 `- 无`。

### 1.8 解析约定（供下游 skill 实现者）

| 解析目标 | 正则 / 定位 |
|---------|------------|
| 所有功能点编号 | 行正则 `^### FP-(\d+)` |
| 功能点名称 | 同上捕获组之后的剩余文本 |
| 所有验收标准 | 行正则 `AC-(\d+)` |
| 验收是否通过 | 匹配 `- \[x\] AC-N` |
| 头部元信息 | 行前缀 `> 状态：` / `> 优先级：` / `> 工作量：` / `> 前置依赖：` |

**返回结构约定**：消费方解析命中后必须以 `{field_name, line, end_line, raw_text}` 结构返回给上层逻辑：
- `field_name`：如 `FP-1` / `AC-3` / `状态` / `接口变更.新增`
- `line` / `end_line`：起止行号（1-indexed）；single-line 字段两者相等
- `raw_text`：原始 markdown 片段（含标题），便于上层重渲染或回写

下游 skill 不得仅返回布尔值或裸字符串，避免位置信息丢失导致回写失误。

## 2 `改动.md` 字段规范

### 2.1 字段顺序（固定，不得乱序）

```
# 改动 - {编号} - {功能名}
> 对应需求：{编号}-{功能名}
> 产出日期：YYYY-MM-DD
> 对应 commit/分支：...

## 功能点清单
## 接口变更
    ### 新增
    ### 修改
    ### 废弃
## 非 HTTP 契约
    ### 定时任务变更
    ### 事件监听变更
    ### 消息通道变更
    ### 邮件/通知通道变更
## 数据库变更
## 配置变更
## 影响面
## 前端交互点
```

**顺序固定**；下游 `test-runner` 按固定章节名 + 固定三级小节名定点解析。

### 2.2 头部元信息

- `> 对应需求：{编号}-{功能名}`：必填，回指子需求目录名。
- `> 产出日期：YYYY-MM-DD`：必填。
- `> 对应 commit/分支：...`：必填；写分支名、HEAD commit hash 或两者组合。

### 2.3 `## 功能点清单`

- 与 `需求.md` 的 `FP-N` 一一对应。
- 格式：

  ```
  - [x] FP-1 功能点名称：一句话描述实现如何满足本 FP
  - [ ] FP-2 功能点名称：未实现，原因：...
  ```

- 未实现（`- [ ]`）时**必须**补充原因（紧跟冒号后）。

### 2.4 `## 接口变更`（关键，三节齐全）

**必须**包含三个三级小节，顺序固定：`### 新增` → `### 修改` → `### 废弃`。任一节无内容时保留小节标题并写 `无`，**不得删除小节**。

#### 接口条目格式（通用）

- **接口标识**：粗体 + inline code，形如 **``POST /api/xxx``**
- 示例：

  ```markdown
  ### 新增
  - **`POST /api/auth/remember-me`**
    - 用途：...
    - 请求：...（body / query / headers）
    - 响应：...（data 结构 / 错误码）
    - 鉴权：...
  ```

  > 注：渲染后呈现为 "**`POST /api/xxx`**"（粗体 + 代码）。

#### `### 新增` 小节

每条接口包含子字段：
- `用途`
- `请求`（body / query / headers / params）
- `响应`（data / error 结构）
- `错误码`
- `鉴权`

缺失写 `无`。

#### `### 修改` 小节

每条接口使用"变更前 / 变更后"对比：

```markdown
- **`GET /api/yyy`**
  - 变更前：...
  - 变更后：...
  - 兼容性：向后兼容 / 破坏性变更（说明迁移方案）
```

#### `### 废弃` 小节

```markdown
- **`DELETE /api/zzz`**
  - 废弃原因：...
  - 替代方案：...
  - 预计下线版本：...
```

#### 解析约定

| 目标 | 定位方式 |
|------|---------|
| 所有接口变更 | 先定位 `## 接口变更` 章节，再枚举三个固定 `###` 子节 |
| 单个接口标识 | 正则 `\*\*\`([A-Z]+)\s+(/\S+)\`\*\*` 捕获 method + path |
| 接口归属（新/改/废） | 当前三级标题决定 |

### 2.4b `## 非 HTTP 契约`（事件 / 定时 / 消息 / 邮件）

某些需求（典型如赛季结算邮件、订单事件广播、定时任务触发）**完全没有 HTTP 接口变更**，但有"非 HTTP 契约"。此时 `## 接口变更` 三节均写 `无`，由本节承接所有非 HTTP 契约。

字段顺序固定，与 `## 接口变更` 类似，但下分四个三级小节（缺类别小节保留并写 `无`）：

```
## 非 HTTP 契约
    ### 定时任务变更
    ### 事件监听变更
    ### 消息通道变更
    ### 邮件/通知通道变更
```

#### `### 定时任务变更`

每条任务条目格式：

```markdown
- **`<任务标识或类名>`**
  - 触发表达式：cron 表达式 / 固定延迟 / 触发器名称
  - 触发条件：业务上"何时跑"（如"赛季结束 30 分钟后"）
  - 业务动作：本次新增/修改的处理逻辑摘要
  - 实现位置：相对路径 + 类/函数名（如 `<backend_dir>/.../SeasonEndJob.java#run`）
```

#### `### 事件监听变更`

```markdown
- **`<事件类型 / EventClass / topic-name>`**
  - 事件来源：发布者
  - 处理器：监听器位置 + 函数
  - 业务动作：摘要
  - 顺序保证：是否需要顺序消费 / 幂等约束
```

#### `### 消息通道变更`

```markdown
- **`<MQ topic / exchange / routing key>`**
  - 协议：RabbitMQ / Kafka / Redis Stream / SQS / NATS / ...
  - 方向：生产 / 消费 / 双向
  - 消费者：位置 + 函数
  - 关键字段：消息体核心字段
```

#### `### 邮件/通知通道变更`

```markdown
- **`<template key 或通知 channel>`**
  - 通道：邮件 / 短信 / 站内信 / IM Webhook / 推送
  - 触发条件：业务摘要
  - 收件规则：单人 / 群发 / TOP-N / 角色筛选
  - 模板位置：路径 + key
  - 实现位置：发送方调用点
```

#### 解析约定

| 目标 | 定位方式 |
|------|---------|
| 是否存在非 HTTP 契约 | 是否含 `## 非 HTTP 契约` 章节 + 任一三级小节非 `无` |
| 单条契约标识 | 本章节内任意 `- \*\*\`(.+?)\`\*\*` 命中 |
| 契约类别（定时/事件/消息/邮件） | 当前三级小节决定 |

**重要**：当 `## 接口变更` 三节全为 `无` 时，**至少要有 `## 非 HTTP 契约`** 中一类非 `无`，否则 `change-summarizer` 应警告"本需求无任何外部可观测变更，是否纯内部重构？"——这是 `test-runner` 兜底自动化策略（见 [`./file-conventions.md`](./file-conventions.md) §9）的输入。

### 2.5 `## 数据库变更`

- 列表项：
  - 表新增 / 字段新增 / 字段修改 / 字段删除
  - 索引新增 / 删除 / 修改
  - 迁移脚本路径（flyway / liquibase / 自建 migration 目录）
- 无数据库变更写 `无`。

### 2.6 `## 配置变更`

- 新增 / 修改 / 删除配置项。
- 每条含 key、默认值、作用、是否需部署时同步配置中心（Nacos / Consul / env）。
- 无配置变更写 `无`。

### 2.7 `## 影响面`

- 被修改的现有功能清单（供 test-runner 生成回归用例）。
- 每条至少含："功能名称 + 受影响原因 + 回归建议"。
- 无影响写 `无`（慎用；大部分改动都有影响面）。

### 2.8 `## 前端交互点`

- 变更涉及的页面 / 组件 / 用户操作路径。
- 每条含："页面或组件 + 操作步骤 + 手测关注点"。
- 无前端变更写 `无`（纯后端接口变更允许）。

## 3 字段缺失与占位约定

### 3.1 生产方（prd-writer / change-summarizer 等）

- **不得删除任何字段**。即使某字段不适用，也必须保留章节标题并写 `无` / `N/A`。
- `无` 与 `N/A` 的区别：
  - `无`：确实没有（如"无外部依赖"）。
  - `N/A`：不适用（如纯后端需求的 `前端交互点` 可写 `N/A`）。
  - 两者可任一使用，但同一项目内建议统一。

### 3.2 消费方（feasibility-assessor / fullstack-builder / test-runner / self-review）

- 解析到字段内容为 `无` / `N/A` → 视为"明确无数据"，跳过该字段的处理。
- 解析到**字段不存在**（章节标题缺失）→ **不得猜测**或自动构造默认值；**必须**向用户反馈并询问：

  ```
  <文件> 中缺失 `<字段名>` 章节（按 schema 应存在）。
  选项：
    【1】由你现在补充内容（回复内容即可）
    【2】由我回填 `无` 并继续（不推荐，下游可能误判）
    【3】回到上游 skill（如 prd-writer / change-summarizer）重写该文件
  ```

- 解析到格式错误（如 FP 编号不符正则 `^### FP-(\d+)`） → 同样 ask 用户修正，不擅自修复。

## 禁止事项

- **不得乱序章节**。字段顺序是下游解析契约的一部分。
- **不得删除字段**。缺失写 `无` / `N/A`。
- **不得私造字段**。如需扩展 schema，应**先**修改本文件并通知所有消费方。
- **消费方不得猜测缺失字段的值**。必须 ask 用户。
- **不得使用非 schema 规定的编号格式**（如用 `功能1` 替代 `FP-1`、用 `验收1` 替代 `AC-1`）。
