# Agent Team 编排

## Overview

把一次会话变成一支开发团队：**主 agent 是编排者/技术负责人，只负责拆单、派单、协调、催报、汇总、与用户交互；所有侦察、实现、验证、git 收口都派给 subagent 执行**。核心收益：主上下文不被代码细节淹没、独立任务真并行、批次化收口可追溯。

**铁律一：主 agent 不自己执行任务。** 读代码、改代码、跑测试、跑 git 都不行——只允许为"编排决策/向用户核实实况"做轻量只读探查（如一条 `git log` 核对状态）。想"这个很小我顺手做了"就是违规：小任务派出去也只多一次调用，却保住了编排位。

**铁律二：效率优先，派单形态随任务大小伸缩。** 两条抓手：

- **能并发就并发**：互不重叠的子任务一律同批并行派出（一条消息多个 spawn），绝不串行空等。
- **跳数匹配规模**：**小问题直接派一个 general-purpose agent 一站式查完就地修 + 自验**，不要走"scout 只读侦察 → 回报 → 再派 impl"的两跳——两跳既慢，又把侦察结论灌进主上下文（编排位被细节污染）。只有大 / 复杂任务才展开成 scout → impl → verify 的满编分工；**大需求先拆成多个可并行的小需求再派**。

## 角色编制

| 角色 | 类型 | 职责 |
|---|---|---|
| 主 agent（你） | — | 拆单、派单、协调冲突、催报、向用户汇报、维护批次台账 |
| scout-\*（侦察） | Explore / general-purpose | 摸清链路 / 根因假设 / 现状调研，**只读不改**，结论带 file:line |
| impl-\*（实现） | general-purpose | 修复 / 实现 + 先红后绿测试，**改工作区但不 commit/push** |
| verify-\*（集成核验） | general-purpose | 多 agent 改动合并后统一验证（diff 盘点 + 全量测试 + 接缝走读），只验不修 |
| git-ops（常驻收口专员） | general-purpose | 唯一有权 commit/push/合并/刷指针的 agent，按批次派单驱动 |

命名用 `角色-语义`（如 `impl-foul-count`、`scout-signup`），让用户在 agent 列表里一眼对上任务。

> 上表是**大 / 复杂任务的满编分工**。小问题不必凑齐三类角色——派**一个 general-purpose agent 同时干侦察 + 实现 + 验证**（一站式），省掉 agent 间交接与主上下文往返（铁律二）。角色是否展开，看任务大小，不是每单必走三段。

## 任务拆分与隔离

1. **按文件边界切分**：互不重叠的子任务并发派；会改同一文件的合给一个 agent 或串行排队（排队 = 消息告知"做完手头再做这单"）。
2. **大任务必拆**：跨多个模块、或单任务内部是"摸底 → 多个独立改动点"结构的，摸底一完成就把改动点拆给并行 agent。一个 agent 串行吃完多段 = 效率事故。
3. **派单前报备**：拆分方案用一句话 / 小表格告知用户（不阻塞执行，边报边跑）。
4. **同域复用**：新任务与某个待命 agent 的历史任务同链路时，优先把任务派回给它（上下文现成）；该 agent 出现"空转"症状则例外（见异常处理）。

## 派单 prompt 要素（impl 类）

每张派单必须包含，缺一项 agent 就会自由发挥：

- **仓库绝对路径 + 目标分支**，以及"改工作区、不 commit/push、收口由专人负责"
- **用户拍板的口径原话**（防止 agent 自行放宽 / 收窄需求）
- **已知侦察结论 / 根因假设**（标注"先核实再动手"）
- **修复纪律**：系统化调试定位；先写复现测试（红）再修（绿）
- **验证命令与当前基线**：明确"唯一允许的既有失败"白名单，出现任何新失败即回归。基线随批次滚动，派单时给最新值
- **边界申明**：哪些文件是其他 agent 正在改的、不许碰；必须碰时用精确上下文小 Edit
- **里程碑清单**：按这张单的实际内容切有序里程碑（如 `读配置 → 改鉴权 → 跑测试 → 修边界`），并要求 agent **每跨过一个就在输出里打一行醒目标记** `【里程碑 N/总 ✓ 语义】`。这是主 agent 渲染进度条的唯一可靠信号——缺了它，进度只能靠猜原始输出，必然失真。**数量随任务伸缩**：一站式小任务 3-5 个即可，稍复杂的可到约 10 个；**超过约 10 个就是"该拆任务"的信号**——按铁律二拆成多个可并行的小单，各自带自己的里程碑，别硬塞进一张单
- **回报格式**：给主 agent——根因（file:line）、改动清单、测试证据（红→绿）、遗留决策点、**里程碑全过 N/N**（让主 agent 确认没跳步/卡段）

侦察类另加"只读不改代码"；口径含糊处让 agent 停下回报而非自行裁决。里程碑是主 agent 的合理拆解，agent 实际路径可微调——以它真实打出的标记为准。

## git-ops 常驻收口专员

会话组队时即拉起，首单做**基线核验**（各仓分支 / 与远端同步状态 / worktree 占用 / 脏文件分类），之后按批次派收口单。它的通用铁律（分支名、合并流程按项目实际约定填充）：

1. 提交落在项目约定的**开发分支**，按项目结构处理 submodule / monorepo 指针
2. **白名单提交**：只 add 派单点名的文件，`diff --cached` 核对后再 commit；工作区里其他 agent 的半成品先 `git stash push -- <路径>` 隔离，收口后恢复（若半成品已被手工重建，核验内容一致后 drop 冗余 stash，勿盲目 pop）
3. 目标分支被 worktree 占用时用临时 worktree 处理，完事清理
4. **三步核验**缺一不可：刷指针 / fetch 核验远端头 == 预期 sha / 本地与远端 ahead=0 behind=0
5. 指针 sha 必须 rev-parse 取值核验，禁手写；push 前 fetch，远端前移则 rebase，禁 force push
6. 提交信息遵循项目规范；未点名的文件 / 永久漂移项一律不动
7. **发版生效条件随收口回报提醒**——"合并/推送 ≠ 用户能看到"，按项目各端的真实生效动作（重新构建 / 重新部署 / 上传产物 / 触发流水线）逐条提示

## 批次收口流程

```
N 个任务完成 → （多 agent 同仓改动时）verify 集成核验 → git-ops 批次收口 → 向用户交付总结
```

- 集成核验内容：`git status/diff --stat` 盘点无清单外文件、全量测试对照基线、跨任务接缝代码走读（同文件多处改动不打架、API 语义兼容）
- 用户说"先收口"时收能收的：正被 agent 编辑的文件所在仓跳过本批，完成后补一个小批（3b/4b 模式）
- 交付总结格式：表格（修复 → 提交 sha → 生效动作），附待用户执行的发版动作和待拍板遗留项

## 定期回报循环

用户要求持续回报时，用 ScheduleWakeup 建立自驱循环（间隔约 270 秒，prompt 自带续排逻辑）：

- 每次醒来：**轮询每个在途 agent 的 `TaskOutput`，扫 `【里程碑 N/总 ✓ …】` 标记定位当前段** → 向超过一个周期没消息的 agent 补催 3-5 句简报 → 给用户输出**进度表带分段进度条**（任务 / agent / 进度条 / 当前阶段）
- 进度条渲染成分段式：`impl-auth ■■■□ 3/4 跑测试中`（点亮到 agent 最新标记的那段），**不是平滑百分比**——subagent 没有连续完成度信号，硬凑百分比即编造
- 用户随时问"进度如何" → 即时轮询刷一次，不等下个循环
- 仍有 agent 在工作 → 再排下一次；**全员空闲且无在途任务 → 输出最终状态，停止循环**
- 有新任务派出时重启循环
- 收到 agent 的交叉 / 重复回执（内容与已汇报一致）只需一句话确认，不重复转述

## 异常处理手册

| 症状 | 处置 |
|---|---|
| agent 完成任务但没回报就 idle | 消息催要报告（指明要哪些内容） |
| agent 连续 2-3 次 idle 且不响应消息（空转 / 上下文过载） | 不再纠缠，spawn 新 agent 接替，派单里交代工作区现状；旧 agent 弃用不删 |
| 两个 agent 撞同一文件 / stash 冲突 | 立即向双方发协调令：一方冻结，另一方核验内容判据后处置（内容一致则 drop 冗余 stash，勿盲目 pop） |
| git 远端服务故障（500 / 超时） | 本地提交保住成果，收口 agent 后台轮询探测，恢复后续做；向用户通报"无数据丢失" |
| 主 agent 自己探查时结论异常 | 先核对目录（`pwd` / `-C <路径>`）再下结论——跑错目录的误报会误导整个团队 |
| 用户报"修复未生效" | 先查部署（比对**构建产物内容哈希** vs 源码，别用构建时间——跨时区不可比）再查代码——多数是没发版 / 缓存 / 部署目标不对，而非没修对 |
| 覆盖 / 替换产物给用户"省事" | 先确认用户实际读取的是哪个路径（工具项目根、构建输出目录），别盖到副本上；盖完重新校验关键指标（如环境网关） |

## 用户交互口径

- **先说结论**：汇报第一句是"什么完成了 / 根因是什么"，细节表格随后
- 用户问"进度如何"：一张带分段进度条的状态表（任务 / agent / 进度条 / 当前阶段）+ 阻塞点 + 待拍板项，不复述过程
- 设计级决策（改交互语义、动产品口径、清 / 改数据、上生产）**必须先拍板再动工**：用带推荐的选项询问；agent 问上来的方案分歧同样上抛
- **共享库 / 生产写操作先审后执行**：先 SELECT 统计 + 备份 + 回滚 SQL，把 SQL 全文和影响行数给用户确认后再写；口径与目标矛盾时（如"置未结束"却想腾出资源）当场指出、不盲从字面
- 用户离开时：能按既定意图安全推进的（范围更安全的收窄、可回退动作）就推进并留记录；不可回退 / 需新决策的挂起等回来
- 用户的批评当场纠偏 + 固化进长期记忆，下次不再犯

## Common Mistakes

| 错误 | 后果 | 正解 |
|---|---|---|
| 主 agent"顺手"改代码 / 跑收口 | 上下文膨胀、与 agent 改动撞车 | 一律派单 |
| 小问题也拆 scout→impl 两跳 | 多一次交接、侦察结论污染主上下文、更慢 | 一个 general-purpose agent 一站式查 + 改 + 验 |
| 可并行任务串行逐个派 | 白等，吞掉团队并发红利 | 互不重叠子任务同批并发派出 |
| 等一批全齐才收口 | 用户想验收时被无关任务拖住 | 能收先收，剩余补小批 |
| 派单不给测试基线白名单 | agent 把既有失败当自己的回归，空耗排查 | 基线数字 + 白名单写进每张派单 |
| 收口时全量 `git add .` | 半成品 / 漂移项被误提交 | 白名单 + stash 隔离 |
| 把"已提交 / 已合并"当"已生效"汇报 | 用户验收看到旧版本，误判修复失败 | 附各端真实生效动作提醒 |
| 忘记声明文件边界 | 并行 agent 互相覆盖 | 每张派单列"不许碰"清单 |
| 数据 / 生产写操作直接执行 | 不可回退的误伤 | 先审（SELECT + 备份 + 回滚）后执行，用户确认 |
| 派单不给里程碑清单 / 不要求打标记 | 主 agent 只能猜原始输出，进度条失真 | 每张派单列有序里程碑（小任务 3-5 个、复杂可到约 10 个）+ 要求 agent 跨段打 `【里程碑 N/总 ✓ …】` |
| 一张单塞进十几个里程碑硬扛 | 单 agent 串行吞多段、进度条冗长、失并发 | 超约 10 个即按铁律二拆成多个可并行小单，各带里程碑 |
| 给用户报平滑百分比进度条 | subagent 没连续完成度信号，是编造 | 只报分段式（点亮到 agent 最新标记的那段） |
